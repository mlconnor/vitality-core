# Food Service Operations Platform - System Design

## Overview

This document outlines the user interface and architecture for an **agentic, intelligence-driven** food service management platform. Rather than traditional CRUD forms and tables, the system acts as an **operational co-pilot** for site managers and chefs.

---

## Core Philosophy: From CRUD to Command Center

The system should:
- Surface what needs attention NOW
- Predict problems before they happen
- Automate routine decisions
- Learn from historical patterns

#### Agentic Features
- **Anomaly Detection**: Auto-identifies unusual cost/waste spikes
- **Root Cause Analysis**: Explains WHY metrics changed
- **Predictive Trends**: "At current trajectory, you'll exceed budget by week 3"
- **Actionable Recommendations**: Specific suggestions, not just data

## Agent Specifications

### 1. Forecast Agent

```
Inputs:   Historical orders, census data, weather, events, day of week
Outputs:  Portion forecasts by recipe, confidence intervals
Methods:  Moving average, exponential smoothing, ML ensemble
Actions: 
  - Generates daily production schedules
  - Adjusts forecasts based on real-time order flow
  - Alerts on high-variance predictions
```

### 2. Procurement Agent

```
Inputs:   Inventory levels, par levels, upcoming menu, forecasts, vendor data
Outputs:  Recommended purchase orders, substitution suggestions
Methods:  Reorder point calculation, economic order quantity, price optimization
Actions:
  - Auto-generates POs when items hit reorder point
  - Suggests vendor switches when better prices available
  - Alerts on delivery delays based on vendor patterns
```

### 3. Compliance Agent

```
Inputs:   Diet orders, allergen data, menu items, meal orders
Outputs:  Validation results, alerts, audit reports
Methods:  Rule-based validation, constraint checking
Actions:
  - Blocks allergen conflicts at order entry
  - Validates all menu items against diet restrictions
  - Generates compliance documentation for inspections
```

### 4. Cost Optimization Agent

```
Inputs:   Recipe costs, current prices, menu, budget targets
Outputs:  Cost alerts, substitution recommendations, budget forecasts
Methods:  Price tracking, margin analysis, what-if modeling
Actions:
  - Alerts when recipe costs exceed targets
  - Suggests ingredient substitutions
  - Recommends menu adjustments to hit budget
```

## Technical Architecture

### Backend Services (Node.js)

```
/api/
  /menu/          - Menu planning, cycle management
  /production/    - Schedules, forecasts, leftovers
  /inventory/     - Stock levels, receiving, issues
  /procurement/   - POs, vendors, specs
  /diners/        - Patient/customer management
  /recipes/       - Recipe CRUD, costing, scaling
  /analytics/     - Reports, KPIs, insights
  /agents/        - AI agent endpoints
```

## Data Flow Examples

### Morning Startup Flow

```
1. Manager opens Command Center
2. Forecast Agent has already generated today's production schedule
3. Compliance Agent flags 2 diet changes from overnight
4. Procurement Agent shows pending deliveries
5. Manager reviews, approves, delegates
```

### Order Entry Flow (Healthcare)

```
1. Staff selects diner from Diner Care
2. System shows today's menu filtered by diner's diet
3. Compliance Agent validates selection against allergies
4. Smart suggestions highlight preferred items
5. Order flows to Production Hub
```

### End-of-Service Flow

```
1. Chef logs leftover counts in Production Hub
2. System calculates waste percentages
3. Cost Agent updates food cost metrics
4. Forecast Agent adjusts future predictions
5. Leftover Intelligence suggests repurposing
```

## Appendix: Database Schema Reference

The system is built on the comprehensive `food_service.ddl` schema which includes:

- **Reference Tables**: Units of measure, food categories, allergens
- **Organization**: Sites, stations, employees
- **Menu Planning**: Meal periods, diet types, cycle menus
- **Recipes**: Ingredients, recipes, recipe_ingredients, menu_items
- **Diners**: Diners, diet_assignments, meal_orders
- **Procurement**: Vendors, product_specifications, purchase_orders
- **Inventory**: Receiving, inventory, storeroom_issues
- **Production**: Production_schedules, forecasts, leftover_reports

See `food_service.ddl` for complete schema documentation.
